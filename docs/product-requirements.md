## 1. 项目背景与业务痛点

电商业务的售后客服是典型的"高并发、低毛利、强情绪、强政策"型场景。基于对中型电商商家客服中心的调研，存在以下普遍痛点：

1. **重复问题占比高，人力成本浪费**
   "我的订单到哪了""可以退款吗""多久能到货"等结构化问题占咨询量的 60% 以上，但仍由人工逐条处理。
2. **夜间无人值守，咨询体验断档**
   非工作时间咨询直接落到留言队列，平均响应时长超过 8 小时，引发投诉与差评。
3. **政策口径不统一**
   不同坐席对同一类售后政策（如 7 天无理由、运费险、价保）给出不一致回复，导致后续争议与升级。
4. **传统 FAQ 机器人无法解决复杂咨询**
   关键词匹配类机器人无法理解"我前天下的订单还没发货，今天能退款吗"这种复合问题，触发转人工率高。
5. **知识沉淀差**
   人工坐席解决的疑难问题没有反哺到机器人能力上，知识库变成"一次性 Word 文档"。
6. **缺少对接业务系统的能力**
   普通对话机器人无法实时查询订单、物流、退款状态，回答停留在"请您稍后联系客服"，价值很低。

本项目要解决的核心问题是：**把客服机器人从"FAQ 检索器"升级为"会查业务系统、会判断风险、会创建工单、会兜底转人工、会沉淀知识"的 Agent 子系统**。

## 2. 项目目标与核心价值主张

### 2.1 总体目标

构建一个**嵌入电商售后业务系统的 AI Agent 子系统**，对接订单、物流、退款、工单、人工坐席与知识库，实现高自动化率、高准确率与可控的人工兜底，并通过知识闭环持续提升能力。

### 2.2 价值主张（一句话）

> 把售后客服 70% 的结构化问题完全自动化，把 30% 的复杂问题以"低风险、可追溯、可升级"的方式交给人工，并让人工解决的每一个 case 自动沉淀回知识库。

### 2.3 核心业务指标目标（上线 3 个月）

| 指标             | 当前基线 | 目标值 | 说明                                                |
| ---------------- | -------- | ------ | --------------------------------------------------- |
| 自动化解决率     | 0%       | ≥ 65%  | Agent 全程闭环，无需人工介入即可结束会话。          |
| 一次解决率（FCR） | 55%      | ≥ 80%  | 一通对话内解决问题，无 24h 内重复咨询。             |
| 平均首响时长     | 90s      | ≤ 3s   | 用户首条消息到 Agent 首个回复 Token 的时间。        |
| 转人工率         | 90%      | ≤ 30%  | 进入会话后最终需要人工接管的比例。                  |
| 错误回答率       | 未量化   | ≤ 2%   | 用户/审核人员标记为"信息错误"或"政策错误"的比例。 |
| 用户满意度（CSAT） | 78%      | ≥ 88%  | 会话结束后用户给出 4-5 星的比例。                   |
| 知识沉淀转化率   | 0%       | ≥ 20%  | 人工解决案例被采纳进知识库 / 人工解决案例总数。     |

## 4. 业务场景清单

下表是 Agent 必须覆盖的 8 大场景。每个场景在 [agent-workflow.md](agent-workflow.md) 都对应一条端到端走查路径。

### 4.1 订单状态咨询

**典型用户输入**："我昨天下的那个订单怎么还没发货？"

**期望行为**：

1. Agent 识别意图为 `order_query`。
2. 抽取槽位：用户 ID（来自登录态）、时间范围（昨天）、订单号（可选）。
3. 调用 `query_order` 工具，按时间范围拉取该用户的订单列表。
4. 命中唯一订单 → 直接回答状态；多笔 → 反问让用户确认。
5. 状态异常（超 48h 未发货）→ 主动提示可申请催单工单。

### 4.2 物流进度查询

**典型用户输入**："订单号 2026052800123 物流到哪了？"

**期望行为**：

1. 意图 = `logistics_query`，槽位 = 订单号。
2. 调用 `query_logistics` 工具，返回最新轨迹节点。
3. 检测到"已签收"但用户仍在问 → 询问是否未收到，决定是否升级为投诉工单。
4. 物流停滞超 72 小时 → 提示用户可创建"物流异常工单"。

### 4.3 退款 / 退货 / 换货咨询

**典型用户输入**："我想退掉一双鞋。"

**期望行为**：

1. 意图 = `refund_query` 或 `refund_apply`。
2. RAG 检索售后政策（7 天无理由、运费规则、价保规则）。
3. 调用 `query_refund` 工具确认订单是否符合条件（是否签收、是否超期、是否享受运费险）。
4. 满足条件 → 调用 `create_refund_ticket` 创建退款工单，返回工单号。
5. 不满足 → 给出原因 + 引用政策文档 + 引导人工。

### 4.4 售后政策咨询（RAG）

**典型用户输入**："你们的价保政策是怎么算的？"

**期望行为**：

1. 意图 = `policy_qa`。
2. 触发 RAG 子图：BM25 + 向量 召回 → Rerank → Top-K。
3. 由 LLM 基于检索片段合成回答，**必须带引用**（文档标题 + 片段编号）。
4. 检索置信度低于阈值 → 走兜底，不允许 LLM 凭空编造。

### 4.5 投诉与情绪升级

**典型用户输入**："你们这态度也太差了，我要投诉！"

**期望行为**：

1. 意图 = `complaint`，情绪 = 负面。
2. **不再尝试自动回答**，直接走 `policy_judge` → 强制 `need_human=true`。
3. 立即创建优先级为"高"的投诉工单，并把上下文摘要写入工单。
4. 通过 `human_handoff` 节点把会话推送到坐席待接队列。
5. 给用户的反馈话术须包含"已为您升级到资深客服"，避免二次激怒。

### 4.6 工单创建与跟踪

**典型用户输入**："上次那个退款工单到哪一步了？"

**期望行为**：

1. 意图 = `ticket_query`。
2. 调用 `query_ticket` 工具，按用户 ID 列出近期工单。
3. 用户进一步追问 → 复述工单当前状态、处理人、最近一次事件。
4. 工单超过 SLA 未处理 → Agent 主动告知并升级。

### 4.7 人工接管与转交

**典型用户输入**：触发条件包括"高风险""低置信度""主动要求人工""情绪激烈"等。

**期望行为**：

1. `policy_judge` 节点根据策略决定 `need_human=true`。
2. `human_handoff` 节点：
   - 锁定会话，禁止 Agent 继续生成；
   - 生成 ≤ 200 字的上下文摘要（用户问题、已查信息、已尝试方案、风险等级）；
   - 推送到坐席待接队列；
   - 给用户反馈"已转人工，预计 X 分钟内回复"。
3. 坐席接管后，Agent 仅作为后台辅助（可被坐席调用查询订单/政策），不直接回复用户。

### 4.8 处理结果回写与知识沉淀

**期望行为**：

1. 工单关闭时，必须回写"解决方案""根因分类""是否可沉淀知识"。
2. 标记为"可沉淀"的工单，由知识运营审核后，通过`knowledge_docs` 后台进入向量索引。
3. 下次相同问题命中新知识 → 完成知识闭环。

## 6. 非功能性需求

### 6.1 性能

- 首 Token 时延（TTFT）≤ 3 秒（P95）。
- 完整回答时延（含工具调用）≤ 8 秒（P95）。
- 单实例并发会话 ≥ 200。

### 6.2 可用性

- 核心链路（对话 / 工单 / 人工接管）SLA ≥ 99.5%。
- LLM 不可用时，Agent 自动降级到"建议转人工 + 创建低优先级工单"。
- Milvus 不可用时，RAG 降级为纯 BM25 检索。

### 6.3 安全与合规

- 用户手机号、地址、订单号在落库、日志、Prompt 中必须脱敏。
- 所有写操作（创建工单、申请退款）需鉴权 + 审计日志。
- Prompt 注入防护：用户输入不能改变系统 Prompt 的行为约束。
- 不允许把真实业务密钥写进代码或仓库，统一从 `.env` 注入。

### 6.4 可观测

- 全链路 `trace_id` 贯穿前端 → 网关 → Agent → 工具 → 日志。
- 关键节点埋点：节点耗时、LLM Token 数、工具调用次数、检索命中数、风险等级。
- 错误码统一，便于前后端联调与监控告警。

### 6.5 可扩展（仅指 MVP 后的演进方向，不在本期实现）

- 节点、工具、Prompt 模板均通过配置或 Python 装饰器注册，新增不改主图代码。
- 多租户隔离、多渠道接入、复杂 RBAC 等扩展项的具体推迟版本号见 [mvp-plan.md §3](mvp-plan.md#3-推迟到后续版本的能力清单明确不做)，本期一律不预留复杂结构。

| 指标          | 口径                                                                                  | 数据源                |
| ------------- | ------------------------------------------------------------------------------------- | --------------------- |

## 8. 范围（In Scope / Out of Scope）

### 8.1 In Scope（本期 = 闭环 MVP）

- **业务闭环**：完整跑通 [mvp-plan.md §2](mvp-plan.md#2-mvp-闭环验收脚本必做) 的 11 步（用户咨询 → 意图识别 → RAG/工具 → 自动回答或工单 → 人工接管 → 工单关闭 → 知识候选 → 审核 → 入库 → 同类问题命中）。
- **角色与渠道**：仅 3 个固定角色（buyer / agent / admin）；仅 Web 渠道。
- **数据**：Mock 订单 / 物流 / 退款（service 层假数据 + MySQL 种子数据）；mock 用户与坐席；20 篇知识种子文档。
- **可观测**：全链路 trace_id + 结构化日志 + 管理端 trace 反查 + 5 个固定指标卡片。
- **部署**：Docker Compose 一键启动；不支持 K8s。

## 9. 里程碑概览

完整的 MVP 推进计划见 [mvp-plan.md](mvp-plan.md)。本期分为 5 个里程碑：

1. **M1**：项目骨架 + 数据库 + 基础 API。
2. **M2**：RAG 子系统（含知识库导入与检索 Debug）。
3. **M3**：LangGraph 主流程 + 基础工具调用。
4. **M4**：工单系统 + 人工接管 + SSE 流式。
5. **M5**：前端工作台 + 可观测 + Docker Compose 一键部署。

## 10. 术语表

| 术语        | 解释                                                                                    |
| ----------- | --------------------------------------------------------------------------------------- |
| Agent       | 本项目特指基于 LangGraph 编排、具备工具调用能力的客服智能体子系统。                       |
| LangGraph   | LangChain 出品的基于状态图的 Agent 编排框架，本项目核心编排层。                           |
| RAG         | Retrieval-Augmented Generation，检索增强生成。                                          |
| Tool Calling | LLM 通过结构化 JSON 调用业务函数的能力。                                                |
| Slot        | 槽位，指 Agent 从用户输入中抽取的结构化字段（如订单号、时间范围）。                       |
| 工单        | 工单系统中的一条处理记录，用于跟踪从用户咨询到问题解决的完整过程。                       |
| 人工接管    | Agent 把会话控制权交给人工坐席的过程，详见 [agent-workflow.md §3.9](agent-workflow.md)。  |
| 知识闭环    | 人工解决案例 → 沉淀 → 审核 → 进入向量库 → 下次自动回答 的循环过程。                       |
| trace_id    | 一次会话或一次调用的全链路追踪 ID，贯穿前后端与第三方调用。                              |
